## Vision

### 10x Check

Instead of treating OAuth as plumbing, build it as the creator's first impression.
The 10x flow: click one button, OAuth redirect, callback, and your profile is already
populated with your creative identity from your social presence. Zero forms filled.
The platform already "knows" you.

Key elements:
- Auto-populate creator profiles from OAuth provider data (avatar, display name, bio)
- Provider-specific field mapping (Google vs Discord vs Twitter all return different shapes)
- CSRF + return-to URL in a single Redis-backed state parameter
- Rate limiting from day 1 (not after the first bot attack)
- Refresh token rotation with JTI-based revocation and grace period
- Structured auth event logging for operational visibility

### Platonic Ideal (aspirational, not all in MVP scope)

The login experience feels invisible. One button click, three seconds, you're in with
a populated profile. Token management is invisible too: silent refresh, multi-device
"just works" via email-based merge. No "session expired" errors. No "link your accounts"
dance. Opening Fugue on your phone with Discord after using Google on your laptop gives
you the same account, seamlessly.

Note: Silent refresh on the frontend (interceptor-based) is aspirational. This plan
implements the backend refresh endpoint; frontend silent refresh is a follow-up task.

### JWT Signing Key

- Algorithm: **HS256** (HMAC-SHA256) with shared secret
- Key source: `JWT_SECRET` env var (minimum 32 bytes, base64-encoded)
- Rotation strategy: deploy new secret alongside old, accept both during transition window (dual-key verification), then remove old. Not in MVP scope, but key storage supports it.
- **Operational note:** HS256 means the same secret is shared across all API server pods. In the K8s deployment (helm/fugue/), `JWT_SECRET` MUST be injected via Kubernetes Secret (not ConfigMap). If the secret leaks, any party can forge tokens. RS256 migration is a future option if the service grows to multiple independent token consumers.

### Infrastructure Setup Task

- Add Redis client initialization with connection pool (max 10 conns) and health check
- Redis is already in `docker-compose.yml` on port 6379
- Connection config via env vars: `REDIS_URL` (default: `redis://localhost:6379`)
- Graceful degradation: if Redis is down, rate limiting is bypassed (fail-open), but refresh token and CSRF state operations fail with 503

## Account Merge Decision Tree

Email-based auto-merge flow when OAuth callback returns user info:

```
OAuth callback receives (provider, provider_id, email, profile)
  │
  ├─ 1. Look up auth_accounts by (provider, provider_id)
  │    └─ FOUND → existing user, issue JWT, done
  │
  ├─ 2. email is NOT NULL → look up within a DB TRANSACTION:
  │    │  BEGIN
  │    │  SELECT creator FROM auth_accounts WHERE email = $1 FOR UPDATE
  │    │  (FOR UPDATE locks existing rows to prevent concurrent merge race)
  │    │
  │    ├─ FOUND → MERGE: add new auth_account to existing creator
  │    │          Profile fields NOT overwritten (first-writer-wins)
  │    │          Log: auth_event{type:"account_merge", provider, creator_id}
  │    │          COMMIT, issue JWT, done
  │    │
  │    └─ NOT FOUND → also check creators.email FOR UPDATE
  │         ├─ FOUND → MERGE (same as above), COMMIT
  │         └─ NOT FOUND → INSERT creator (email, nickname, avatar, roles='{}', contacts='{}')
  │                        ON CONFLICT (email) DO NOTHING + re-SELECT
  │                        (handles race where two concurrent INSERTs hit the UNIQUE constraint)
  │                        INSERT auth_account for this creator
  │                        COMMIT, issue JWT, done
  │
  └─ 3. email IS NULL (e.g., Twitter without email scope)
       └─ CREATE new creator with email=NULL, roles='{}', contacts='{}'
          + auth_account
          Auto-fill nickname, avatar_url from provider mapping
          If this creator later logs in with a provider that HAS email:
            → Step 2 will NOT find a match (because creators.email is NULL)
            → A second creator record is created
            → Manual merge needed (out of MVP scope, add to TODOS.md)
```

**Race condition handling:**
- `FOR UPDATE` locks existing rows during concurrent merge attempts
- For truly-new creators (no existing row), rely on `creators.email` UNIQUE constraint
  with `ON CONFLICT DO NOTHING` + re-query pattern. This ensures exactly one creator
  per email regardless of concurrency.

## Cookie Specification

| Cookie | Value | httpOnly | Secure | SameSite | Path | Max-Age |
|--------|-------|----------|--------|----------|------|---------|
| `fugue_access` | JWT access token | YES | YES (skip in dev) | Lax | `/` | 900 (15 min) |
| `fugue_refresh` | JWT refresh token | YES | YES (skip in dev) | Lax | `/api/auth` | 604800 (7 days) |

- Refresh cookie path restricted to `/api/auth` so it's only sent on auth endpoints
- Domain: derived from `FRONTEND_URL` env var
- On logout: both cookies cleared with Max-Age=0

### Token Delivery on Callback

OAuth providers redirect via GET to the callback URL. The callback handler:
1. Receives GET request with `code` and `state` query params
2. Exchanges code for provider access token (server-side)
3. Fetches user profile from provider
4. Runs account merge/create logic
5. Issues JWT access + refresh tokens
6. Sets httpOnly cookies on the response
7. Redirects (302) to `return_to` URL (from CSRF state) or `{FRONTEND_URL}/`

The frontend does NOT receive tokens in the response body. Tokens are delivered
exclusively via httpOnly cookies for the web client. For API clients (future),
a separate `POST /api/auth/token` endpoint can be added.

**Note:** The callback endpoint is `GET /api/auth/{provider}/callback` (not POST),
since OAuth providers redirect via GET.

## Rate Limiting Specification

| Endpoint | Limit | Window | Response |
|----------|-------|--------|----------|
| `GET /api/auth/{provider}/login` | 10 req | 1 min / IP | 429 + `Retry-After: 60` |
| `GET /api/auth/{provider}/callback` | 5 req | 1 min / IP | 429 + `Retry-After: 60` |
| `POST /api/auth/refresh` | 20 req | 1 min / IP | 429 + `Retry-After: 60` |
| `POST /api/auth/logout` | 10 req | 1 min / IP | 429 + `Retry-After: 60` |

Implementation: Redis INCR + EXPIRE (fixed-window counter). Key format: `rl:{endpoint}:{ip}`.
Fixed-window is sufficient for auth endpoints. Sliding window (sorted sets) is overkill here.

## Refresh Token Rotation Flow

**Redis key design:**
- `rt:{jti}` → JSON `{"creator_id": "uuid", "created_at": "ISO8601", "status": "active|rotated"}`
- `rt_index:{creator_id}` → Redis SET of active JTIs for this creator

**Flow:**
1. Client sends refresh token (via cookie) to `POST /api/auth/refresh`
2. Server validates refresh token JWT, extracts JTI
3. Server checks Redis key `rt:{jti}` exists and status is "active" or "rotated" (within grace)
4. Server issues new access token + new refresh token (new JTI)
5. Server stores `rt:{new_jti}` in Redis with 7-day TTL, adds to `rt_index:{creator_id}`
6. Server sets `rt:{old_jti}` status to "rotated" with 10-second TTL
7. Server removes old JTI from `rt_index:{creator_id}`
8. If old refresh token is reused AFTER grace period (key deleted): **compromise detection**
   - Read `rt_index:{creator_id}` to get all active JTIs
   - Delete each `rt:{jti}` key
   - Delete `rt_index:{creator_id}`
   - Log: auth_event{type:"token_compromise_detected", creator_id}
   - Return 401 "Session revoked for security"

Grace period handles legitimate concurrent requests (e.g., two tabs refreshing simultaneously).

```json
  "error": "error message (only on failure events)"

```

## CSRF State Flow

1. `GET /api/auth/{provider}/login`:
   - Generate random state token (32 bytes, base64url)
   - Store in Redis: `oauth_state:{state}` → `{provider, return_to, created_at}` with 10-min TTL
   - `return_to` extracted from `?redirect=` query param (validated: must start with `/`, no absolute URLs)
   - Redirect (302) to provider's authorization URL with state param
2. `GET /api/auth/{provider}/callback`:
   - Extract `state` from callback query params
   - Look up `oauth_state:{state}` in Redis
   - If not found or expired → redirect to `{FRONTEND_URL}/login?error=invalid_state`
   - Delete the state key (single-use)
   - Proceed with token exchange
   - After JWT issuance, set cookies and redirect to `return_to` (or `{FRONTEND_URL}/`)

## JWT Auth Middleware

Token extraction order:
1. Check `fugue_access` httpOnly cookie
2. If no cookie, check `Authorization: Bearer {token}` header
3. If neither present → 401 Unauthorized

Middleware validates: signature (HS256), expiration, issuer claim.
On valid token: extract `creator_id` from `sub` claim, set in request context.
On expired token: return 401 with `X-Token-Expired: true` header (frontend uses this to trigger refresh).

## Reviewer Concerns (from spec review, 3 iterations, score 5→6→8)

Remaining issues not fully resolved:

1. **Lock ordering in concurrent merge:** Always acquire `creators` row lock before `auth_accounts` to prevent deadlocks.
2. **Compromise detection revokes ALL sessions:** When a rotated refresh token is reused after grace period, all devices are logged out. Frontend should handle 401 "Session revoked for security" by redirecting to login with explanatory message.
3. **Redis-down = login fully broken:** "Graceful degradation" only applies to rate limiting (fail-open). OAuth login and token refresh require Redis and will return 503.
4. **Dev mode detection for Secure cookie:** Dev mode is detected when `OAUTH_CALLBACK_BASE_URL` starts with `http://` (not `https://`). In dev mode, Secure flag is omitted.
5. **`creators.email` population:** The `email` from OAuth provider response maps to `creators.email` on creator creation. This is implicit in the decision tree but should be explicit in implementation.
